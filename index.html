문제의식

대학생 굿즈(과잠·키링·텀블러)는 수요가 분명한데도 디자인의 벽, 소량 제작 단가, 공동구매 운영의 번거로움에 막힙니다. 두띵은 '디자인 → 펀딩 → 제작'을 한 흐름으로 잇습니다. 기술적으로 가장 까다로웠던 지점은 ① AI 이미지 생성이 한 번에 60초를 넘겨 CDN 타임아웃에 걸리는 것, ② 결제 마감 스케줄러가 상시 안전하게 돌아야 하는 것, ③ 흰 제품에 디자인을 입힐 실루엣 마스크가 부정확한 것이었습니다.

주요 기능 · 나의 기여

  • AI 이미지 생성 서비스 단독 구현 — gpt-image-1로 옷 사진을 제작용 도면으로 합성하고 모델 착용 가상 피팅까지 생성. 입력 크기 캡(8MB)·60초 중복요청 캐시·일일 한도·과금 호출 로그·타임아웃(AbortController)까지 비용·안정성 5겹 통제를 서비스에 내장
  • AI 공급자 추상화 — ImageAiService / TextAiService 인터페이스를 두고 구현만 교체해 Gemini → OpenAI를 라우트 변경 없이 무중단 전환
  • 비동기 작업 + 폴링 파이프라인 — 생성을 즉시 202(jobId)로 응답하고, 본인 작업만 3초 간격으로 폴링해 CDN 타임아웃을 우회
  • 제품 실루엣 마스크 Python 툴체인 — numpy·scipy·Pillow 형태학 연산 + rembg 누끼로 흰 제품에서 디자인 영역만 추출(뚜껑·모자·고리 제외), 키링 10종 포함 제품별 정밀 마스크
  • 인증·로깅 하드닝 — Google OAuth2 state를 httpOnly 쿠키에 바인딩해 로그인 CSRF 차단, 전 /api 액세스 로그 + 감사 로그(본 흐름 비차단)
  • DB 스키마·배포 — 마이그레이션 45개 중 37개 작성, terser·clean-css 압축 후 S3 sync + CloudFront 무효화를 묶은 배포 파이프라인
  • AI 도구 기반 코드 감사 — Claude Code·CodeRabbit·Kiro로 OWASP·DB·리뷰 관점 라운드를 20여 회 반복(커밋 251건 중 228건이 Claude 페어)

기술 의사결정 (Why AWS)

  • CloudFront — Google OAuth는 HTTPS 리다이렉트만 허용 → ACM 무료 TLS가 로그인의 전제. 정적·이미지는 엣지 캐싱으로 EC2 부하를 덜고 단일 진입점 라우팅
  • EC2(PM2) > Lambda — 60초 주기 결제 마감 스케줄러가 상시 구동돼야 해, 요청 기반 Lambda보다 장기 실행 프로세스가 자연스러움
  • RDS PostgreSQL — user↔펀딩↔주문이 강한 관계형이고 결제 정합성에 ACID 트랜잭션·집계 쿼리가 필요해 관리형 RDB 선택

트러블슈팅

70초 AI 생성 vs 60초 CloudFront 오리진 타임아웃
문제

원인
동기 단일 요청이라 생성이 끝날 때까지 한 연결을 붙잡아 60초를 못 버팀
해결
검증 직후 jobId를 202로 반환, 생성은 백그라운드, GET /ai/jobs/:id를 본인 작업만 3초 간격·최대 200초 폴링으로 회수
결과
60초 벽을 넘겨 AI 생성이 타임아웃 없이 안정적으로 완료
마감 스케줄러가 DB 페일오버 시 죽거나 마감을 중복 처리
해결
pg_try_advisory_lock non-blocking 분산락으로 락 미획득 시 이번 tick을 건너뛰고, connect 실패도 흡수해 크래시 없이 다음 tick에서 회복
결과
DB 단절 시 프로세스 크래시·마감 중복 처리 모두 방지

주요 기능 · 나의 기여

  • 대중교통 + 보행자 경로 통합 — ODsay searchPubTransPath + TMAP 보행자 경로를 매핑하고 route_summary_json을 TTL 캐시(hit/miss/forceRefresh/stale 4단계). 보행 구간은 TMAP GeoJSON으로 인도 곡선을 채우고 실패 시 합성 직선으로 graceful fallback
  • 경로 장애 자동복구 메커니즘 — 실패를 transient/auth/mapping으로 분류하는 RouteRefreshOutcome enum, 등록 시 1회 backoff 재시도 + RouteRetryScheduler가 PENDING_RETRY를 5분 쿨다운·최대 5회 keyset 페이지네이션으로 per-row 독립 트랜잭션 재호출, 401/403은 운영자 신호화
  • 권장 출발 시각 · 웹 푸시 — 권장출발 = 도착 − 소요로 산출, ±3분 윈도우로 EARLIER/ON_TIME/LATER 안내, ONCE/DAILY/WEEKLY/CUSTOM 다음 일정을 KST로 계산, PushScheduler가 per-iteration 격리로 VAPID 웹 푸시 발송
  • 지오코딩 / 역지오코딩 — Kakao keyword+address 머지 검색, query를 NFC·lowercase 정규화 후 SHA-256으로 30일 캐시, miss 캐시로 quota 보호, UPSERT race는 REQUIRES_NEW + 1회 retry로 격리
  • Flyway V1–V10 마이그레이션 — 모든 스키마 변경을 nullable ADD COLUMN으로 backward-compatible하게 버전 관리

트러블슈팅

외부 경로 API 실패 시 경로가 잠긴 채 회복 수단이 없어 알림 누락
문제
ODsay가 일시 장애(5xx)나 키 만료(401/403)로 실패하면 경로가 PENDING_RETRY로 잠겨, 사용자가 직접 수정하지 않으면 영구 정체
원인
실패를 구분하지 않고 모두 같게 처리 — 재시도해도 소용없는 4xx와 의미 있는 5xx가 섞임
해결
RouteRefreshOutcome enum으로 분류, 등록 시 1회 즉시 재시도 + 스케줄러가 5분 쿨다운·최대 5회 배치 재시도, 401/403은 운영자에 노출
결과
사용자 개입 없이 일시 장애가 약 25분 내 자동 복구. 단위 12건 + 통합 회귀 가드
데모 직전 500 — 요일 6개 이상이면 일정 등록 실패
원인
요일 CSV 컬럼이 VARCHAR(20). 평일 5요일(19자)은 통과하지만 6요일 이상은 길이를 넘겨 'Data too long' — 특정 요일이 아니라 '요일 개수'가 진짜 트리거
해결
V10 마이그레이션으로 VARCHAR(31) 확장(MySQL INSTANT 무중단) + DTO에 @Size(max=7) 검증을 더해 컬럼 폭과 입력을 함께 bound (defense in depth)
결과
6요일 이상 정상 등록. 통합 테스트 3건(201/400/400)으로 재발 차단
